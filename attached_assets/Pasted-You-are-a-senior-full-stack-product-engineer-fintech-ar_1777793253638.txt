You are a senior full-stack product engineer, fintech architect, SME accounting workflow expert, reconciliation system designer, UX designer, and SaaS product builder.

Build a complete, working, mobile-first web application called “ParchiFlow”.

ParchiFlow is a Universal SME Ledger Intelligence, Collection Automation, and Reconciliation OS for Indian SMEs.

The product should help small retailers, wholesalers, traders, service businesses, contractors, manufacturers, distributors, pharmacies, local vendors, and unorganized SMEs convert messy business money data into one simple dashboard.

The users may currently operate through:
- Paper parchi
- Bahi-khata
- Memory
- Verbal credit
- WhatsApp messages
- Excel sheets
- Bank statements
- UPI payments
- Tally
- BUSY
- Marg
- Accountant-managed ledgers

The product should answer the business owner’s most important questions:

1. Kis se kitna lena hai?
2. Kis ko kitna dena hai?
3. Kaun overdue hai?
4. Kaun payment promise karke nahi de raha?
5. Kaunsa payment receive hua but match nahi hua?
6. Kaunsa transaction duplicate lag raha hai?
7. Kaunsa transaction doubtful hai?
8. Kisko WhatsApp reminder bhejna hai?
9. Aaj kitna collect hua?
10. Mere Tally/BUSY/Marg/Excel data ka simple mobile view kya hai?

This is NOT traditional accounting software.
This is NOT another Tally.
This is NOT only a Khatabook clone.
This is NOT only a CredFlow clone.
This is NOT only a Biz Analyst clone.

This is:

Digital Parchi
+ Party Ledger
+ Accounting Software Import Layer
+ Bank/UPI Money Signal Layer
+ Reconciliation Engine
+ WhatsApp Reminder Layer

The system should feel like:

========================
PRODUCT PRINCIPLES
========================

1. Zero accounting knowledge required.
2. Mobile-first.
3. Every core action should take less than 5 seconds.
4. Use simple Indian SME language:
   - Lena Hai
   - Dena Hai
   - Paisa Mila
   - Paisa Diya
   - Udhaar Diya
   - Advance Liya
   - Advance Diya
   - Follow-up
   - Reminder
   - Doubtful
   - Match Karna Hai
5. Avoid heavy accounting jargon in the frontend.
6. Backend may use proper accounting logic, but UI must remain simple.
7. Never blindly update final ledger for doubtful transactions.
8. Every imported or detected money event must pass through matching and reconciliation logic.
9. Manual parchi, accounting software data, bank/UPI data, WhatsApp proof, and payment gateway data should all become “money events”.
10. The final ledger should update only after:
    - High-confidence auto-match, or
    - User confirmation, or
    - Manual approval.
11. MVP must be honest:
    - Real Tally/BUSY/Marg live sync may require connector/API/local agent.
    - SMS/notification reading requires native app and permissions.
    - WhatsApp automation requires WhatsApp Business API approval.
    - Payment auto-settlement requires payment gateway integration.
12. Build a working MVP, not a design mockup.

========================
HIGH-LEVEL WORKFLOW
========================

The complete product workflow is:

1. User logs in with mobile number.
2. User creates business profile.
3. User selects data source:
   - Manual parchi
   - Excel
   - CSV
   - Bank statement
   - Tally
   - BUSY
   - Marg
   - Payment gateway
4. User adds/imports data.
5. System converts all incoming data into universal money events.
6. System normalizes money events into:
   - Parties
   - Ledger entries
   - Outstandings
   - Payments
7. System checks:
   - Duplicate transaction
   - Duplicate party
   - Unknown party
   - UTR/reference match
   - Invoice match
   - Amount match
   - Date match
   - Source conflict
   - Suspicious/doubtful transaction
8. System assigns confidence score.
9. If high confidence, auto-match but log it.
10. If medium/low confidence, send to reconciliation queue.
11. User reviews reconciliation queue.
12. Confirmed transactions update:
   - Party ledger
   - Outstanding
   - Dashboard
   - Follow-up list
13. User sends WhatsApp reminder.
14. User records payment or imports bank/UPI payment.
15. System matches payment with outstanding.
16. Dashboard updates.
17. Reports become available.

MODULE 14: PAYMENT MATCHING ENGINE
========================

- Bank statement
- UPI narration

- Payment gateway mock
- WhatsApp payment claim

- Party
- Invoice
- Outstanding
- Amount
- UTR/reference
- Date
- Narration
- Mobile/GSTIN

Matching score:
- Same UTR/reference: +50
- Same invoice number: +40
- Same party mobile/GSTIN: +25
- Same amount: +20
- Date within 1 day: +15
- Narration keyword match: +10
- Same external source ID: +50

Decision:
- 85–100: Auto-match but log it
- 60–84: Suggested match, user confirmation required
- 30–59: Reconciliation queue
- 0–29: Suspense/unmatched

Payment cases:
- Exact payment
- Partial payment
- Excess payment
- Short payment
- Unknown credit
- Duplicate payment
- Gateway + bank duplicate
- Screenshot claim but no bank credit
- Cash payment manually entered
- Cheque pending
- Cheque bounced
- Refund/reversal

MODULE 15: RECONCILIATION ENGINE
========================

This is one of the most important modules.

No doubtful transaction should directly update final ledger without review.

Every incoming money event must pass through:

1. Source identification
2. Duplicate detection

MODULE 16: RECONCILIATION EDGE CASES
========================

Handle these edge cases:

Duplicate cases:
- Same UTR already exists
- Same invoice imported from Tally and Excel
- Same amount/date/party entered manually and from bank
- Payment gateway webhook and bank credit both appear
- Staff and owner both record same payment
- WhatsApp screenshot later matched with bank credit
- Same party with spelling variation

Doubtful cases:
- Amount received but payer unknown
- Customer says paid but no bank credit
- Bank credit received but no invoice due
- Payment amount does not match invoice
- Same UPI reference appears twice
- Date mismatch between bank and ledger
- Cash payment entered without party
- Expense without category
- Vendor payment without bill
- Cheque deposited but not cleared
- Cheque bounced later
- Payment reversed/refunded
- GST invoice cancelled but ledger still open
- Credit note exists against invoice
- Debit note exists against vendor
- Opening balance differs across sources

Matching outcomes:
- Auto matched
- Suggested match
- Partial match
- Split required
- Suspense
- Duplicate
- Dispute
- Ignored
- Reversed
